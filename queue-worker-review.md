# Queue Worker Code Review

- **Missing BullMQ provider import blocks DI** — `WebhooksService` injects `BullQueueService` (apps/backend/src/webhooks/webhooks.service.ts), but `WebhooksModule` only imports `GitHubModule` and `DashboardModule` (apps/backend/src/webhooks/webhooks.module.ts:5). Without importing `BullmqModule`, Nest can't resolve `BullQueueService`, so webhook handling will fail to bootstrap or enqueue jobs.
- **Duplicate `SyncWorkerService` registrations create multiple instances** — The class is provided/exported in both `SyncModule` (apps/backend/src/sync/sync.module.ts:6) and `WorkersModule` (apps/backend/src/workers/workers.module.ts:7). Importing both modules (as done by `BullmqModule`) spawns separate `SyncWorkerService` instances and makes DI resolution order-dependent, risking divergent state/logging and harder-to-trace worker behavior.
- **BullMQ module imports amplify the duplication risk** — `BullmqModule` imports `forwardRef(() => SyncModule)` and `forwardRef(() => WorkersModule)` (apps/backend/src/bullmq/bullmq.module.ts:9), even though `WorkersModule` already imports `SyncModule`. Pulling in both modules alongside the duplicate provider above increases the chance of inconsistent `SyncWorkerService` instances and circular dependency headaches.
